Section 1–2

Key Concepts

The foundational ideas from the talk — what AI agents are, how they connect to SQL Server, and why the distinction matters for DBAs.

🤖

AI Agent ≠ Chatbot

A chatbot generates text. An agent takes actions — it reasons, plans multi-step operations, calls tools (like running SQL queries), evaluates results, and loops until a goal is met. Different risk profile entirely.

🔗

Model Context Protocol (MCP)

An open standard that defines how agents discover and call tools. Think of it as an API contract: tool name, parameters, permissions. Self-hostable on-prem. Your control surface for agent access.

🛡️

Defense in Depth

Four layers: Network & Identity → MCP Protocol Guardrails → SQL Server Security (RLS, audit, least-privilege) → Application Logic (human-in-the-loop, approval gates). Each layer assumes the one above might fail.

🔄

Start Read-Only, Earn Writes

Begin with SELECT-only access via approved views. Graduate to stored procedure execution. Writes require DBA approval gates. Most agent value comes from reading and analyzing data, not modifying it.

This MCP tool definition restricts the agent to read-only queries on a specific database. The agent cannot execute any DML or DDL.

  "name": "query_sql",
  "description": "Execute a read-only SQL query",
  "inputSchema": {
    "type": "object",
    "properties": {
      "query": {
        "type": "string",
        "description": "SELECT statement only"
      },
      "database": {
        "type": "string",
        "enum": ["AdventureWorks", "Analytics"]
      }
    },
    "required": ["query", "database"]
  },
  "permissions": ["db_datareader"],
  "guardrails": {
    "blocked_keywords": ["INSERT", "UPDATE", "DELETE", "DROP", "ALTER", "EXEC"],
    "max_rows": 10000,
    "timeout_seconds": 30
  }
}

Create a dedicated, least-privilege login for your MCP agent. Never use sa or sysadmin. Scope to specific schemas.

T-SQL 
-- Create a dedicated agent login
CREATE LOGIN [AgentServiceAccount]
  WITH PASSWORD = '<strong-password>',
  CHECK_POLICY = ON;

-- Map to database with minimal permissions
USE AdventureWorks;
CREATE USER [AgentServiceAccount]
  FOR LOGIN [AgentServiceAccount];

-- Read-only on specific schemas
GRANT SELECT ON SCHEMA::Sales
  TO [AgentServiceAccount];
GRANT SELECT ON SCHEMA::Production
  TO [AgentServiceAccount];

-- Deny everything dangerous
DENY INSERT, UPDATE, DELETE, EXECUTE
  ON SCHEMA::Sales TO [AgentServiceAccount];
DENY ALTER, CONTROL
  ON SCHEMA::Sales TO [AgentServiceAccount];

Capture every query the agent runs with SQL Server Audit. Correlate with MCP server logs for full traceability.

T-SQL 
-- Create a server-level audit
CREATE SERVER AUDIT [AgentAudit]
  TO FILE (
    FILEPATH = 'C:\AuditLogs\Agent\',
    MAXSIZE = 100 MB,
    MAX_ROLLOVER_FILES = 10
  )
  WITH (ON_FAILURE = CONTINUE);

-- Enable the audit
ALTER SERVER AUDIT [AgentAudit]
  WITH (STATE = ON);

-- Audit specification for agent user
CREATE DATABASE AUDIT SPECIFICATION [AgentActions]
  FOR SERVER AUDIT [AgentAudit]
  ADD (SELECT, INSERT, UPDATE, DELETE
    ON SCHEMA::Sales
    BY [AgentServiceAccount])
  WITH (STATE = ON);

Query the audit log to see exactly what the agent did, when, and how many rows it touched.

T-SQL 
-- View recent agent activity
SELECT
    event_time,
    action_id,
    server_principal_name,
    database_name,
    schema_name,
    object_name,
    statement,
    affected_rows
FROM sys.fn_get_audit_file(
    'C:\AuditLogs\Agent\*.sqlaudit',
    NULL, NULL
)
WHERE server_principal_name = 'AgentServiceAccount'
  AND event_time >= DATEADD(HOUR, -1, GETUTCDATE())
ORDER BY event_time DESC;
Compliance

Compliance Checklist

Before deploying any agent with SQL Server access in a regulated environment, verify each of these items.

  • Dedicated service account — Agent uses a unique, least-privilege SQL login. No shared credentials. No sa access.
  • Read-only by default — MCP tools configured for SELECT only. DML/DDL blocked at the protocol layer.
  • SQL Server Audit enabled — Every agent query captured with timestamp, statement text, row count, and user identity.
  • Network segmentation — MCP server on private endpoint or VPN. No public internet exposure for agent-to-database traffic.
  • Schema restrictions — Agent access limited to specific schemas. No access to system tables, master, or tempdb.
  • Row-Level Security — RLS policies active on sensitive tables. Agent sees only data appropriate to its security context.
  • Human-in-the-loop for writes — Any data modification requires DBA approval. No autonomous writes to production.
  • Prompt injection awareness — Input validation in MCP layer. Blocked keywords. Parameterized queries where possible.
  • Rate limiting — Agent queries capped per minute/hour. Prevents runaway loops and resource exhaustion.
  • Log correlation — MCP server logs, application logs, and SQL Server Audit files are correlated by session ID for forensics.
